# ZenPulse 腕带硬件报告 — 从设计到落地

> **项目**：ZenPulse 中医 AI 诊断系统（毕设）  
> **版本**：v1.0  
> **日期**：2026-Q2  
> **关联文档**：[硬件开发指南.md](./硬件开发指南.md)、[脉象硬件腕带方案.md](./脉象硬件腕带方案.md)、[脉象算法开发文档.md](./脉象算法开发文档.md)

## 摘要

本报告描述 ZenPulse 腕式生理参数采集硬件从需求分析、方案选型、电路与结构设计、固件开发到与小程序/后端联调的完整过程。系统采用 **双 MAX30102 光学传感器 + ESP32-S3 主控**，通过 **BLE GATT** 向微信小程序推送 IR 原始序列，由后端提取心率与近似血氧，参与 AI 辨证流程。

硬件定位是 **毕设可演示原型**，非医疗级认证设备。产品对外表述为「光学传感器估算心率/血氧」，不输出中医脉名（迟脉/数脉等）。

## 1. 需求与目标

### 1.1 业务需求

| 需求 | 说明 |
|------|------|
| 叙事一致 | 腕部桡动脉采集，符合「切诊」场景，避免指夹式方案 |
| 可演示 | 手机小程序 10 s 内完成采集并显示 HR/SpO₂ 状态 |
| 可降级 | 蓝牙失败时可手填 HR，不影响后半段望诊与辨证 |
| 可解释 | 算法边界写入 `limitations`，答辩时可说明非医疗级 |
| 可扩展 | 预留 IMU、PCB 量产、WiFi 调试等多条演进路径 |

### 1.2 技术指标（v1.0 达成）

| 指标 | 目标 | 实现 |
|------|------|------|
| 采样率 | 100 Hz | `SAMPLE_HZ = 100`，固件定时读 FIFO |
| 双通道 | CH1 + CH2 独立 IR | I2C 双地址 0x57 / 0xAE |
| 最短有效窗口 | ≥ 5 s / ≥ 500 点 | 后端质控 + 小程序 10 s 采集 |
| 传输 | 手机直连 | BLE Notify，广播名 `ZenPulse` |
| 调试通道 | 实验室 CSV | WiFi TCP :8080 保留 |
| 本地反馈 | 佩戴状态 | 可选 0.96" OLED（BLE/信号/HR） |

### 1.3 能力分级

| 等级 | 内容 | 落地状态 |
|------|------|----------|
| **L1** | 双 MAX30102 + ESP32 + 腕带 + BLE | ✅ 主路径 |
| **L2** | + MPU6050 运动检测 | ⏳ 固件已预留 `ENABLE_IMU` |
| **L3** | 寸关尺 / 浮沉 / 指压分级 | ❌ 需专用机械，非本期 |

## 2. 方案选型

### 2.1 传感器

**选用 MAX30102**（I2C 心率/血氧模块）：

- 集成 RED/IR LED 与光电接收，适合 PPG 脉搏波采集  
- 模块化采购成本低（约 ¥8–15/颗），便于毕设迭代  
- 双颗并联可做多点冗余与平均降噪  

**未选用**：指夹式成品（叙事不符）；专业医疗 SpO₂（成本与认证超出范围）。

### 2.2 主控

**选用 ESP32-S3-DevKitC-1**（ESP32-S3-WROOM-1）：

- 内置 **BLE 4.2**，满足小程序 GATT 直连  
- 原生 USB CDC，烧录与串口调试方便  
- WiFi 并存，保留 TCP 实验室采集路径  
- I2C 400 kHz 下可稳定驱动双 MAX30102 + OLED  

经典 ESP32-WROOM-32 仍可通过 `USE_ESP32_S3=0` 兼容（GPIO21/22 I2C）。

### 2.3 传输方式演进

| 阶段 | 方式 | 用途 |
|------|------|------|
| H1 台架 | WiFi TCP :8080 | `pulse_collector.py`、示波器式调试 |
| **v1.0 产品** | **BLE GATT Notify** | 微信小程序 `bleVitals.js` 主路径 |
| 未来 | WebSocket 实时流 | 文档标注为扩展项 |

## 3. 系统设计

### 3.1 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                    腕带硬件（佩戴侧）                          │
│  ┌─────────────────┐         ┌─────────────────────────┐   │
│  │ Sensor 区域      │  I2C   │ Main 区域（腕背外侧）    │   │
│  │ MAX30102 #1 #2  │◄──────►│ ESP32-S3                │   │
│  │ 遮光 O 圈 · 贴皮肤 │        │ OLED（可选）· BLE 天线   │   │
│  └─────────────────┘         └───────────┬─────────────┘   │
│           3D 托板 + 魔术贴腕带              │ BLE            │
└───────────────────────────────────────────┼─────────────────┘
                                            ▼
                              ┌─────────────────────────┐
                              │ 微信小程序               │
                              │ bleVitals → vitals API  │
                              └───────────┬─────────────┘
                                          ▼
                              ┌─────────────────────────┐
                              │ FastAPI 后端             │
                              │ VitalsService 信号处理   │
                              └─────────────────────────┘
```

### 3.2 电路原理

```
                    ┌──────────────────┐
   USB 5V ─────────►│  ESP32-S3        │
                    │  3.3V LDO        ├─── 3.3V ──┬── MAX30102 #1 (0x57)
                    │  GPIO8  SDA      ├───────────┼── MAX30102 #2 (0xAE)
                    │  GPIO9  SCL      ├───────────┼── SSD1306 OLED (0x3C)
                    │  BLE / WiFi      ├───────────┼── MPU6050 (0x68) [L2]
                    └──────────────────┘           └── GND 共地
```

**设计要点**：

- 所有模块并联同一 I2C 总线，地址不冲突  
- 第二颗 MAX30102 须 **改 SDO 焊盘** 至 0xAE，否则双器件无法共存  
- **禁止** GPIO 直供传感器 VIN；走开发板 3.3V 电源轨  
- 线长 < 15 cm，远离 USB 开关噪声  

### 3.3 I2C 地址表

| 器件 | 7-bit 地址 | 备注 |
|------|------------|------|
| MAX30102 #1 | 0x57 | SDO → GND（默认） |
| MAX30102 #2 | 0xAE | SDO → VCC |
| SSD1306 OLED | 0x3C | 与 PPG 共总线 |
| MPU6050 | 0x68 | L2 可选 |

验证工具：`firmware/tools/i2c_scanner/i2c_scanner.ino`

## 4. 机械结构设计

### 4.1 设计目标

- 传感器 LED 窗口对准 **桡骨茎突内侧桡动脉**  
- 腕带可调（周长 14–22 cm），松紧以「可塞进一根手指」为宜  
- 遮光结构抑制环境光对 IR 的干扰  
- 主控与天线位于 **腕背外侧**，避免被皮肤遮挡 BLE 信号  

### 4.2 取脉布局

| 传感器 | 位置 | 固件通道 |
|--------|------|----------|
| #1 | 关/寸部，弧底对准桡动脉 | CH1 |
| #2 | 沿动脉向近心 8–12 mm | CH2 |

用户佩戴说明：坐位、前臂与心脏同高、测量期间保持静止 10 s。

### 4.3 3D 托板规格

详见 `hardware/wrist_bracket/SPEC.md`，核心参数：

| 参数 | 值 |
|------|-----|
| 外形 | 52 × 34 × 9 mm |
| 底面弧半径 | R40 mm（贴合腕背） |
| 传感器间距 | 12 mm（中心距） |
| LED 窗口 | Ø5.5 mm + 2 mm 遮光裙边 |
| 腕带槽 | 宽 22 mm，适配 20 mm 魔术贴 |
| 材料 | PLA / PETG |

参数化模型：`hardware/wrist_bracket/bracket.scad`

### 4.4 装配流程

1. MAX30102 ×2 用 M2 固定于托板底面，窗口对准弧面  
2. 每颗传感器贴 **黑色硅胶 O 圈** 遮光  
3. 1 mm 海绵垫缓冲皮肤接触  
4. ESP32-S3 固定于托板顶面或外侧小盒（I2C 线 ≤ 15 cm）  
5. 可选 OLED 装于主控仓顶面  
6. 魔术贴穿过两侧槽，**传感器区避开魔术贴覆盖**

## 5. 固件设计

### 5.1 软件架构

**路径**：`firmware/esp32_pulse_wrist/`

| 模块 | 文件 | 职责 |
|------|------|------|
| 主循环 | `esp32_pulse_wrist.ino` | 100 Hz 采样、WiFi/TCP、命令解析 |
| BLE 传输 | `ble_transport.cpp` | GATT 服务、Notify 推流 |
| 显示 | `display_ui.cpp` | SSD1306 状态刷新（4 Hz） |
| 配置 | `config.h` | WiFi、引脚、功能开关 |

依赖库：SparkFun MAX3010x、ESP32 BLE Arduino、U8g2（OLED）。

### 5.2 采样逻辑

```c
#define SAMPLE_HZ 100
// 每 10 ms：
//   ir1 = sensor1.getIR();
//   ir2 = sensor2.getIR();
//   bleTransportEmitSample(ir1, ir2);  // 不依赖 TCP 连接
```

MAX30102 配置（`config.example.h`）：

- LED 亮度：0x24  
- 样本平均：4  
- 脉冲宽度：411 µs  

### 5.3 通信协议

**BLE GATT（v1.0 主路径）**

| 项 | 值 |
|----|-----|
| 广播名 | `ZenPulse` |
| Service UUID | `0000fff0-0000-1000-8000-00805f9b34fb` |
| Notify UUID | `0000fff1-0000-1000-8000-00805f9b34fb` |
| 载荷 | `CH1:{ir1},CH2:{ir2}\n` |

**WiFi TCP（调试保留）**

- 端口：8080  
- 格式：与 BLE 相同（v0/v1）  
- 命令：`STREAM` / `CAPTURE 20` / `PING`  

**v2 扩展（ENABLE_IMU=1）**

```
TS:{ms},CH1:{ir1},CH2:{ir2},AX:{ax},AY:{ay},AZ:{az}\n
```

### 5.4 功能开关

| 宏 | 默认 | 说明 |
|----|------|------|
| `ENABLE_BLE` | 1 | 小程序采集必需 |
| `ENABLE_OLED` | 1 | 本地佩戴反馈 |
| `ENABLE_IMU` | 0 | L2 运动质控 |
| `USE_ESP32_S3` | 1 | S3 引脚 8/9；经典 ESP32 改 0 |

### 5.5 烧录与验证

```bash
cd firmware/esp32_pulse_wrist
cp config.example.h config.h   # 填写 WiFi（TCP 调试时）
pio run -e esp32-s3-devkitc-1 -t upload
pio device monitor             # 115200，应见 [OK] BLE GATT ZenPulse
```

台架验收：手指覆盖传感器，串口每秒打印 `IR1=… IR2=…`，随脉搏周期性变化。

## 6. 软件联调（落地路径）

### 6.1 v1.0 端到端数据流

```
ESP32 固件 (100 Hz)
    │ BLE Notify: CH1/CH2
    ▼
wechat-miniprogram/utils/bleVitals.js  (~10 s 采集，双通道对齐)
    │ POST /api/vitals/analyze
    ▼
VitalsService → extractor（双通道合并、带通、峰值）→ rules（阈值映射）
    │ vitals_assessment
    ▼
metrics 页展示 → upload 页 diagnose（透传 assessment，不重传波形）
    ▼
result 页生理参数卡片
```

### 6.2 实验室调试路径（保留）

```bash
python3 scripts/hardware/pulse_collector.py \
  --host <ESP32_IP> --port 8080 \
  --duration 30 --out data/pulse_capture.csv
```

可用于 CSV 存档、算法标定；**答辩主演示走 BLE + 小程序**。

### 6.3 质控与告警

| 层级 | 机制 |
|------|------|
| 固件 | OLED 显示信号 OK（IR 范围 50k–260k） |
| 小程序 | CH1/CH2 长度不一致时对齐并提示 |
| 后端 | 采样不足 422；双通道 mismatch 写入 alerts；quality_score 降权 |

## 8. 测试与验收

### 8.1 硬件验收标准

| 项 | 合格标准 |
|----|----------|
| I2C | 扫描到 0x57 与 0xAE 两个地址 |
| 静止 IR | 波形周期性，非平线 |
| 双通道 | CH1 与 CH2 相关但不完全相同 |
| 心率 | 与血氧仪/手动触脉对比误差约 ±5 bpm |
| BLE | 手机可搜到 `ZenPulse`，Notify 有数据 |
| 遮光 | 强光下遮光套明显改善波形 |

### 8.2 系统验收（答辩用）

- [ ] 固件烧录后手机能搜到 `ZenPulse`  
- [ ] 小程序采集 10 s 后 HR/SpO₂ 有合理数值  
- [ ] 诊断结果页展示 `vitals_assessment`，无脉象类型  
- [ ] `limitations` 文案可见  
- [ ] 蓝牙失败时可手填 HR 完成演示  

### 8.3 答辩演示步骤

1. 笔记本：`python3 web_server.py`  
2. `wechat-miniprogram/config/env.js` 配置局域网 IP  
3. 微信开发者工具 → 真机预览 → 不校验合法域名  
4. 演示：**体征（蓝牙或手填）→ 望诊 → 结果页**  
5. 备用：预录操作视频；蓝牙失败改手填 HR

## 9. PCB 量产规划（设计完成，未打板）

`hardware/zenpulse_pcb/` 已输出 **双板 + FPC** 方案，供后续迭代：

| 板卡 | 尺寸 | 层数 | 内容 |
|------|------|------|------|
| **Main PCB** | 38×28 mm | 4 层 | ESP32-S3、OLED、TP4056 充电、LiPo、BLE 天线 |
| **Sensor PCB** | 34×16 mm | 2 层 | 双 MAX30102、FPC 8P 连接 |

设计原则：传感器贴皮肤、主控与天线在外侧，避免射频被遮挡。引脚分配与现固件 GPIO8/9 一致，便于软件迁移。

## 10. 已知局限与后续工作

### 10.1 当前局限

1. **SpO₂ 为单通道 IR 启发式**，非双波长比值法，不可与医疗血氧仪对比认证  
2. **未实现运动伪影剔除**（L2 IMU 已预留）  
3. **不测血压**，血压仍由用户手填  
4. **开发板 + 杜邦线** 方案，体积与可靠性不及 PCB 量产版  
5. 真机 BLE 依赖微信基础库、系统蓝牙权限与局域网配置  

### 10.2 后续扩展

| 方向 | 说明 |
|------|------|
| L2 运动质控 | 启用 MPU6050 + 滑动窗拒识 |
| 双波长 SpO₂ | MAX30102 Red + IR 标定系数 |
| PCB 打板 | 按 zenpulse_pcb 指南嘉立创下单 |
| 锂电续航 | TP4056 + DW01 方案已在 PCB 文档中 |
| Web Bluetooth | Web 端直连（非当前范围） |

## 11. 结论

ZenPulse 腕带硬件已完成 **从方案设计到毕设可演示落地** 的全链路：

- **设计层**：明确 L1 双通道 PPG + ESP32-S3 + BLE 架构，机械规格与 PCB 演进路线清晰  
- **实现层**：固件支持 100 Hz 双通道采样、BLE/TCP 双传输、可选 OLED 本地反馈  
- **集成层**：与微信小程序、`/api/vitals/analyze` 及诊断链路打通，具备降级与质控  
- **文档层**：BOM、接线、烧录、验收、答辩步骤均可复现  

当前形态为 **开发板 + 3D 打印托带原型**，满足毕设答辩「实物 + 软件闭环」要求；PCB 量产与 L2/L3 能力作为明确后续项，不影响 v1.0 演示完整性。
